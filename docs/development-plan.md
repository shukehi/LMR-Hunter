# 开发计划

## 1. 文档目的

本文档定义 LMR-Hunter 的研发阶段、优先级、交付物、验收标准和当前下一步动作。

本文件回答的是“先做什么，做到什么程度才允许进入下一阶段”。

当前阶段状态以 [项目状态](status.md) 为准；本文件主要维护研发步骤和阶段门禁，不重复维护运行态摘要。

本文件中的“阶段 0 - 4”表示产品生命周期阶段，不等同于 [审查记录](reviews/index.md) 中的各轮优化/修复阶段编号。

## 2. 研发阶段

每个阶段拆分为可独立编码和验证的子步骤。每一步有明确的输入依赖、交付物和完成标志。验收步骤（标记为 `✓`）是阶段门禁，必须全部通过才能进入下一阶段。

- 通用写入接口（支持批量写入、失败重试、写入计数）
- `pydantic` 数据模型定义（原始事件、信号、订单等）

完成标志：

- 所有表可创建、可写入、可查询
- 写入接口有单元测试覆盖
- 数据模型通过类型校验

#### 步骤 1.2：行情接入 — 强平流与成交流

依赖：步骤 1.1

目标：接入核心行情数据并落库

交付物：

- `gateway/` 模块
- Binance WebSocket 连接管理（`forceOrder` 强平流、`aggTrade` 成交流）

完成标志：

- 连接可建立，数据可解析并写入 SQLite
- 双时间戳字段完整
- 可手动运行 10 分钟无报错

#### 步骤 1.3：行情接入 — K 线流与重连逻辑

依赖：步骤 1.2

目标：补全 VWAP 所需的 K 线数据源，实现断线自动恢复

交付物：

- K 线 WebSocket 接入（`kline_1m`）
- 自动重连逻辑（指数退避、最大重试次数、重连后数据补齐）
- 重连事件写入 `service_heartbeats`
- 启动时通过 REST 拉取最近 15 根 K 线，保证 VWAP 冷启动

完成标志：

- K 线数据可持续写入
- 手动断网后 30 秒内自动重连并恢复数据流
- 重连事件有日志和落库记录

#### 步骤 1.4：特征计算引擎

依赖：步骤 1.3

目标：基于已落库数据计算策略所需特征，包括修复路径观测指标

交付物：

- `features/` 模块
- 滑动窗口强平聚合（`liq_window_sec` 内的总名义金额）
- 强平加速率计算（当前窗口 vs 前一窗口的金额变化率，供衰减层使用）
- `VWAP_15M` 计算（基于最近 15 根 1m K 线的 Rolling VWAP）
- 价格偏离率计算（当前价格 vs `VWAP_15M`，单位 bps）
- 修复路径观测指标（Observe Mode 阶段持续采集）：
  - 每次冲击后的回弹启动延迟（最低点到首次回弹 > 10bps 的时间）
  - 回弹深度（回弹幅度占冲击幅度的比例）
  - 永续-现货（指数）价差快照（冲击时和修复时）
- 特征计算结果可写入 `signals` 表的中间字段

完成标志：

- 强平聚合值与手动计算一致
- 强平加速率在已知序列下输出正确
- VWAP_15M 与 K 线数据交叉验证正确
- 偏离率计算在已知输入下输出正确
- 冷启动后 VWAP 在 15 根 K 线到齐后开始输出
- 回弹观测指标在冲击事件后可自动计算并落库

#### 步骤 1.5：监控与健康检查

- 基本告警：连接断开、写库失败、延迟异常

完成标志：

- 心跳定时写入
- 延迟统计可输出到日志
- 告警条件可触发并记录

#### 步骤 2.1：信号引擎

依赖：步骤 1.4

目标：组合事件层、偏离层和衰减层，输出可执行信号

交付物：

- `signal/` 模块
- 事件层判定：滑动窗口内强平总额 ≥ `liq_notional_threshold` 且方向为多头被平
- 偏离层判定：当前价格偏离 `VWAP_15M` ≥ `deviation_threshold_bps`
- 衰减层判定：当前窗口强平金额 ≤ 前窗口 × `liq_acceleration_ratio`（强平未加速）
- 信号对象生成（含完整审计字段：触发时间、强平总额、方向、当前价、基准价、偏离率、强平加速率）
- 信号写入 `signals` 表

完成标志：

- 三层信号（事件 + 偏离 + 衰减）在满足条件时正确触发
- 任一层不满足时无误触发
- 衰减层在强平加速时正确阻断信号
- 信号对象包含所有架构文档要求的审计字段
- 可用历史数据回放验证信号触发逻辑

#### 步骤 2.2：影子撮合引擎

依赖：步骤 2.1

目标：模拟挂单和成交判定，含延迟惩罚

交付物：

- `shadow/` 模块核心：虚拟挂单生成器 + 撮合判定器
- 挂单价计算：信号触发价 - `entry_offset_bps`
- 延迟惩罚：信号在 `Exchange Time T` 触发，虚拟挂单生效时间不早于 `T + estimated_latency_ms`
- 成交判定：仅当 `T + estimated_latency_ms` 之后的市场数据中，成交价触及挂单价，才视为成交
- 超时判定：`order_ttl_sec` 内未成交则标记为 `EXPIRED`
- 虚拟订单写入 `shadow_orders` 表

完成标志：

- 延迟惩罚逻辑正确，无前视偏差
- 超时撤单逻辑正确
- 可用已知数据构造”应成交”和”不应成交”的测试用例并通过
- 每笔虚拟订单有完整的时间戳链路（信号时间 → 生效时间 → 成交/过期时间）

**注意：这是阶段 2 复杂度最高的步骤，建议预留充足时间。**

#### 步骤 2.3：影子持仓管理

依赖：步骤 2.2

目标：模拟成交后的止盈、止损和时间止损

交付物：

- 影子持仓追踪器（虚拟成交后创建虚拟持仓）
- 均值回归止盈：价格回归至 `VWAP_15M` 附近时平仓
- 价格硬止损：成交后继续下跌 `hard_stop_bps` 时平仓
- 时间止损：成交后 `time_stop_sec` 内未修复时平仓
- 每笔虚拟持仓记录：最大有利偏移（MFE）、最大不利偏移（MAE）、退出原因、模拟盈亏
- 影子审计扩展字段：若挂单更近/更远的假设结果

完成标志：

- 三种退出路径（止盈、价格止损、时间止损）各有测试用例
- MFE/MAE 计算正确
- 模拟盈亏 = 退出价 - 入场价 - 模拟手续费
- 手续费按 Maker 入场 + Maker/Taker 退出分别计算

#### 步骤 2.4：多级建仓模拟

依赖：步骤 2.3

目标：验证分批建仓在低延迟下是否优于单笔建仓

交付物：

- 多级建仓模拟器（2-3 级，每级独立撮合判定）
- 与单笔模式的对比报表（胜率、盈亏比、平均入场成本）
- 可配置参数：`entry_levels`、`entry_step_bps`、每级仓位比例

完成标志：

- 多级建仓在延迟 < 100ms 下正确执行
- 多级建仓在延迟 > 100ms 下自动退化为单笔模式
- 对比报表可输出，字段完整

#### 步骤 2.5：回放脚本与参数研究工具

依赖：步骤 2.3（可与 2.4 并行）

目标：将影子数据转化为可分析的研究材料

交付物：

- 历史数据回放脚本（读取 SQLite → 重跑信号+影子撮合 → 输出结果）
- 参数扫描工具（对 `deviation_threshold_bps`、`entry_offset_bps`、`hard_stop_bps` 等做网格搜索）
- 影子结果报表生成器（总信号数、成交率、胜率、盈亏比、净期望值、手续费占比）
- 竞争压力观测报表（回弹启动延迟、回弹深度的时间趋势）

完成标志：

- 回放脚本可重跑已有数据，结果与实时影子模式一致
- 参数扫描可输出不同参数组合下的盈亏矩阵
- 报表包含策略文档要求的所有指标

#### 步骤 2.✓：阶段 2 验收与参数冻结

依赖：步骤 2.1 - 2.5 全部完成

验收流程：

1. 在 VPS 上以 Shadow Mode 连续运行至少 `14` 个自然日
2. 运行结束后生成完整影子报表
3. 使用参数扫描工具确定第一版参数
4. 冻结第一版参数，记录冻结理由

验收标准：

- 已积累至少 `100` 个合格信号和至少 `30` 笔虚拟成交
- 计入手续费后，预估净期望值 > `0`
- 至少 `95%` 的信号样本具备完整审计字段
- 无前视偏差（所有虚拟成交发生在 `T + latency_ms` 之后）
- 多级建仓 vs 单笔建仓的对比结论已记录
- 竞争压力指标无趋势性恶化
- 第一版参数已冻结并写入策略文档

**决策点：若净期望值 ≤ 0 且参数扫描无法找到正期望组合，应暂停项目并重新评估策略可行性。**

#### 步骤 3.1：风控引擎

依赖：步骤 2.1（信号引擎已可用）

目标：实现预交易风控审核、账户级保护和自动化策略失效检测

交付物：

- `risk/` 模块
- 预交易风控检查链（状态检查 → 持仓检查 → 挂单冲突检查 → 日损检查 → 保证金检查）
- 日损累计与熔断逻辑
- 连续亏损暂停逻辑（`5` 笔连亏 → `COOLDOWN`）
- 冷静期计时器（止损后 `30` 分钟）
- 自动化策略失效检测（策略文档 6.1 节定义的 5 项指标）：
  - 滚动胜率崩塌检测（最近 20 笔胜率 < 30% → `HALTED`）
  - 回弹深度/延迟趋势恶化检测（7 天滚动对比 → 告警）
  - 影子-实盘偏差检测（相关系数 < 0.5 连续 14 天 → `HALTED`）
  - 连续最大回撤检测（权益回撤 > 10% → `HALTED`）
- 风控事件写入 `risk_events` 表

完成标志：

- 每项预交易检查有独立的通过/拒绝测试用例
- 超限下单被 `100%` 阻断
- 风控状态变迁（`WATCHING` → `COOLDOWN` → `HALTED`）可正确触发
- 5 项失效检测指标可正确计算并在触发时自动暂停
- 自动暂停后系统拒绝自动恢复，必须人工介入
- 所有风控事件有完整的日志和落库记录

#### 步骤 3.2：执行引擎与 OMS

依赖：步骤 3.1

目标：接入 Binance 测试网，实现完整下单/撤单/回报链路

交付物：

- `execution/` 模块
- Binance Futures Testnet API 接入（REST 下单/撤单 + WebSocket 用户流）
- 订单状态机实现（`SIGNAL_READY` → `ORDER_PENDING` → `ORDER_FILLED` / `CANCELLED` / `REJECTED` / `EXPIRED`）
- `PARTIALLY_FILLED` 处理：立即撤余单，已成交部分独立管理
- Post-Only 入场约束 + 止损主动退出双路径
- 订单幂等去重（`clientOrderId` 唯一性）
- 订单和成交写入 `live_orders` 和 `fills` 表

完成标志：

- 可在测试网完成一次完整的”信号 → 风控 → 下单 → 成交 → 止盈/止损 → 平仓”生命周期
- Post-Only 被拒时正确处理（不重试为 Taker）
- 幂等去重在重复信号下不产生重复订单
- 所有订单状态变迁有日志和落库

**注意：这是阶段 3 复杂度最高的步骤。**

#### 步骤 3.3：状态恢复与重启对账

依赖：步骤 3.2

目标：系统重启后可安全恢复到正确状态

交付物：

- 重启对账流程：暂停 → `GET /fapi/v2/positionRisk` → `GET /fapi/v1/openOrders` → 与本地 SQLite 对账 → 修正差异 → 恢复 `WATCHING`
- 僵尸订单识别与清理（交易所有但本地无的订单）
- 孤儿持仓识别与告警（交易所有但本地无的持仓）
- 对账不可调和时自动进入 `HALTED`

完成标志：

- 手动 kill 进程后重启，可正确恢复状态
- 有持仓时重启，持仓被正确识别并继续管理
- 有挂单时重启，挂单被正确识别或清理
- 对账失败时系统进入 `HALTED` 而非继续交易

#### 步骤 3.✓：阶段 3 验收

依赖：步骤 3.1 - 3.3 全部完成

验收流程：

1. 在测试网上执行完整生命周期演练
2. 模拟各种异常场景（重启、断网、部分成交、风控触发）

验收标准：

- 已完成至少 `20` 次完整订单生命周期演练
- 已完成至少 `3` 次强制重启恢复演练，且 `0` 次重复订单、`0` 次孤儿订单
- 风控对超限下单的阻断成功率为 `100%`
- `PARTIALLY_FILLED → 撤余单 → 管理已成交部分` 路径已验证
- 主动止损退出路径在测试网中正确落账

### 阶段 4：小仓位实盘

本阶段无需拆分，核心工作是配置切换和持续观测。

目标：

- 用最小仓位验证实盘行为
- 对比影子模式与实盘偏差

交付物：

- 实盘配置（保证金 ≤ 5% 权益、杠杆 ≤ 3x、最大挂单 = 1、最大持仓 = 1）
- 实盘审计报表
- 影子 vs 实盘偏差分析报告

验收标准：

- 首次实盘阶段至少运行 `14` 个自然日或完成 `10` 笔实盘成交，以较晚者为准
- `0` 次重复订单、`0` 次孤儿订单、`0` 次风控绕过事件
- `100%` 的入场订单保持 `Post-Only` 约束，除非止损设计明确允许主动退出
- 实盘阶段未触发超出文档定义的仓位与杠杆上限

**决策点：若实盘净期望值 ≤ 0 或与影子模式长期显著背离，应暂停并重新评估。**
