# 策略文档

## 1. 文档目的

本文档定义 LMR-Hunter 的策略边界、核心命题、交易假设、信号逻辑和参数体系。

本文件回答的是“为什么做、何时做、何时不做”。

## 2. 策略概述

### 2.1 策略名称

`清算驱动的均值回归策略`

### 2.2 策略目标

识别 BTC 永续合约中由大规模强制平仓引发的短期流动性真空，并通过更深位置的被动限价单，参与价格修复后的均值回归。

### 2.3 策略立场

该策略不是趋势预测模型，不基于形态、指标崇拜或主观盘感。

它只关心三个事实：

1. 是否出现了异常强平事件
2. 价格是否被强平冲击到了短期成本中心之外
3. 当前市场是否仍具备修复条件

## 3. 使用场景与约束

本策略的初始设计假设面向以下约束：

- 账户规模较小（~1000 USD）
- 具备编程能力
- 部署于东京 VPS，端到端延迟约 `50-80ms`
- 不与 HFT 做速度竞争，但利用低延迟提升成交率和止损精度
- 接受低胜率、高赔率
- 日常可投入时间有限

这意味着策略必须满足：

- 主要依赖自动化执行
- 入场以被动限价单为主，低延迟场景下可使用条件市价单作为补充
- 必须用严格风控覆盖错误事件
- 止损利用低延迟优势实现精确退出

## 4. 策略命题

策略基于以下可检验命题：

1. 杠杆头寸被强制平仓时，会产生非自愿、非理性的市价抛售。
2. 当强平规模足够大时，短时卖压会击穿正常盘口承接，形成流动性真空。
3. 并非所有超跌都会修复，但部分极端事件会在数秒到数分钟内出现修复性回抽。
4. 个人小资金无法在”第一时间”取胜，但可以在更深位置等待后续成交。

### 4.1 修复机制的因果解释

命题 3 是策略最核心的假设。”价格偏离后会修复”不是自然规律，必须有具体的买方和具体的动机。以下是已知的修复驱动力：

**驱动力 A：永续-现货套利者**

- 强平冲击主要发生在永续合约，现货市场受影响较小
- 当永续价格被砸到显著低于现货价格时，永续-现货价差扩大
- 套利者买入永续、卖出现货，赚取价差收敛利润
- 该驱动力的强度取决于永续 vs 现货价差的大小

**驱动力 B：做市商重新报价**

- 强平冲击期间，做市商会临时撤回买单以规避被动成交风险
- 冲击结束后，做市商重新在较低价位挂出买单，提供新的承接层
- 做市商的回归速度取决于冲击是否已经衰减、波动率是否回落

**驱动力 C：其他策略的买入信号**

- 价格急跌会触发其他交易者的均值回归、超跌反弹等策略的买入信号
- 技术指标（如 RSI 超卖）也会吸引主观交易者入场
- 该驱动力的强度不可预测，但在高流动性市场（BTC）中通常存在

**驱动力 D：强平卖压自然衰减**

- 强平是有限的：当杠杆头寸被清算完毕，卖压消失
- 若无新的基本面利空，失去卖压的市场倾向于回升
- 该驱动力的前提是”强平已经结束”，而非”正在强平中”

### 4.2 修复失败的条件

修复不会发生的情况：

- **基本面驱动的下跌**：价格下跌原因不是杠杆踩踏而是真实利空（监管打击、交易所暴雷等），公允价值本身下移，没有”偏离”可修复
- **连环清算未结束**：第一波强平触发了更多止损和更多强平，卖压在加速而非衰减
- **流动性枯竭**：做市商和套利者判断风险过高，拒绝回归市场
- **市场结构性转变**：恐慌情绪导致所有参与者同方向抛售，无人愿意提供对手盘

### 4.3 对信号设计的指导意义

修复机制分析直接指向两个信号缺陷和一个观测要求：

1. **缺陷：当前信号未判断”强平是否在衰减”** — 驱动力 B 和 D 都依赖冲击已经结束或正在结束，但事件层只统计”强平有多大”，没有判断”强平在加速还是减速”
2. **缺陷：当前信号未观测永续-现货价差** — 驱动力 A 的强度可通过价差直接度量，但首版信号未纳入
3. **观测要求：Observe Mode 必须记录每次冲击后的实际修复路径** — 包括修复延迟、修复幅度、修复驱动力的间接指标（价差、盘口恢复速度）

缺陷 1 在 MVP 中通过新增”强平衰减层”部分解决（见 7.5 节）。缺陷 2 作为 P3 增强项保留，不阻塞 MVP。

## 5. 交易假设

项目当前采用以下交易假设：

- 交易所提供稳定且可用的强平事件流
- BTC 永续合约长期存在高杠杆交易者
- 极端强平后的部分价格偏离具有修复性
- 小账户的深度挂单不会引起明显冲击

这些是假设，不是事实。必须通过后续观察模式和影子模式验证。

## 6. 失效前提

以下情况会直接削弱或破坏策略前提：

- 价格下跌来自重大基本面崩塌，而非单纯杠杆踩踏
- 强平规模仍在持续放大，没有进入衰减阶段
- 盘口真实承接远弱于估计
- 成交后价格继续单边加速下跌，没有修复
- 实盘行为和影子模式长期显著背离

### 6.1 自动化失效检测

策略失效不能依赖人工观察，必须有自动化检测机制。以下指标应在 Shadow Mode 和 Live Mode 中持续计算，并在触发阈值时自动暂停策略：

| 失效指标 | 计算方式 | 阈值（首版） | 动作 |
|----------|----------|-------------|------|
| 滚动胜率崩塌 | 最近 20 笔成交的胜率 | < 30% | 自动暂停，进入 `HALTED` |
| 回弹深度趋势恶化 | 最近 7 天 vs 前 7 天的平均回弹深度 | 下降 > 50% | 告警 + 人工评估 |
| 回弹延迟趋势恶化 | 最近 7 天 vs 前 7 天的平均回弹启动延迟 | 增加 > 100% | 告警 + 人工评估 |
| 影子-实盘偏差 | 影子模式预测盈亏 vs 实盘实际盈亏的相关性 | 相关系数 < 0.5（连续 14 天） | 自动暂停 |
| 连续最大回撤 | 账户权益从峰值回撤比例 | > 10% | 自动暂停，进入 `HALTED` |

阈值均为首版猜测值，需根据 Shadow Mode 数据校准。

自动暂停后的恢复必须经过人工评估，不允许自动恢复。

## 7. 信号结构

系统信号分为五层：

MVP 冻结决策如下：

- `事件层`、`偏离层` 和 `衰减层` 是首版硬门槛
- `流动性层` 不是 MVP 硬依赖，而是 `P3` 增强过滤器
- 所有下单请求都必须先经过预交易风控审核

### 7.1 事件层

判断是否出现异常强平事件。

最小输入：

- 强平方向
- 强平名义金额
- 滑动窗口聚合结果

### 7.2 偏离层

判断价格是否已经显著偏离短期成本中心。

候选基准：

- `VWAP_15M`
- `EMA_15M`

MVP 已冻结使用 `VWAP_15M` 作为唯一基准价。

### 7.3 衰减层（MVP 新增）

判断强平卖压是否正在衰减，而非仍在加速。

该层解决的问题：事件层只告诉你"强平够大了"，但没有告诉你"强平快结束了"。在强平加速阶段入场，等于在瀑布中间接刀。

最小输入：

- 最近两个连续窗口（`liq_window_sec`）的强平名义金额
- 当前窗口 vs 前一窗口的金额变化率

判定逻辑：

- 若当前窗口强平金额 > 前一窗口 × `liq_acceleration_ratio`，判定为"强平加速中"，**阻断入场**
- 若当前窗口强平金额 ≤ 前一窗口，判定为"强平衰减"，允许通过
- `liq_acceleration_ratio` 是可配置参数，首版默认值为 `1.0`（即只要当前窗口不大于前一窗口就允许）

该层是 MVP 硬门槛，与事件层和偏离层并列。

### 7.4 流动性层（P3 增强项）

判断当前盘口是否存在足够明显的承接真空。

该层在 `P3` 之前不作为阻断下单的硬门槛。

在 MVP 中：

- 若有可用深度数据，可作为观测字段落库
- 若无可用深度数据，不阻塞首版信号生成
- 首版信号默认仅基于”事件层 + 偏离层 + 衰减层 + 预交易风控”

最小输入：

- 当前盘口深度摘要
- 指定价格范围内的买盘总量
- 强平名义金额与买盘深度的对比

### 7.5 永续-现货价差层（P3 增强项）

度量驱动力 A（永续-现货套利）的强度。

在 MVP 中不实现，但 Observe Mode 应开始采集以下数据：

- BTC 永续合约价格 vs 现货（或指数）价格的价差
- 强平事件发生时的价差快照
- 修复发生时的价差变化

该数据用于后续判断”价差是否足够大以吸引套利者介入”。

### 7.6 执行与风控层

在允许进入下单逻辑之前，必须先通过预交易风控审核。

MVP 的默认准入链路为：

1. `事件层` 满足
2. `偏离层` 满足
3. `衰减层` 满足（强平未加速）
4. 若启用 `流动性层`，则其结果作为附加过滤器
5. 预交易风控审核通过
6. 才允许生成可执行订单意图

预交易风控至少检查以下条件：

- 当前无持仓
- 当前无冲突订单
- 风控未进入暂停或熔断状态
- 未超过单日亏损上限
- 未超过单笔保证金限制
- 未超过最大挂单数限制

## 8. 入场逻辑

满足触发条件后，执行以下入场逻辑：

### 8.1 主入场模式：被动限价单

1. 计算理论挂单价
2. 将理论价格转换为符合交易所精度的实际挂单价
3. 以 `Post-Only LIMIT` 方式挂出买单
4. 为该订单设置有效期
5. 若超时未成交，则撤单并进入观察状态

### 8.2 补充入场模式：多级建仓（低延迟解锁）

当端到端延迟 < 100ms 时，可启用多级建仓模式：

1. 信号触发后，第 1 单以 `30%` 仓位在触发价附近挂出
2. 若价格继续下跌超过 `entry_step_bps`，第 2 单以 `30%` 仓位在更低价位挂出
3. 若价格继续下跌超过 `2 × entry_step_bps`，第 3 单以 `40%` 仓位在最低价位挂出
4. 分批建仓降低平均入场成本

多级建仓的前提条件：

- 总仓位不超过单笔保证金上限
- 每级下单间隔由实际延迟决定（延迟 > 100ms 时退化为单笔模式）
- 所有分级订单共享同一止损和止盈逻辑

### 8.3 入场纪律

本策略禁止使用市价追涨或主动吃单反弹。多级建仓不是加仓，而是在同一事件窗口内分批寻找更优入场位。

## 9. 出场逻辑

出场分为三类：

### 9.1 均值回归止盈

当价格修复至以下任一目标时，触发止盈：

- 回归短期基准价
- 到达预设回补区间

`P3` 之后如启用分批止盈，再增加分批目标位。

### 9.2 时间止损

若成交后在规定时间内未出现修复，应退出仓位。

该逻辑用于识别“这不是短时踩踏，而是更深层的趋势性重估”。

### 9.3 价格止损

若成交后继续跌破最大允许不利区间，应立即止损。

## 10. 初版限制

MVP 明确禁止以下行为：

- 追价买入
- 无限制补仓
- 同时持有多个方向仓位
- 用人工临时改逻辑覆盖系统信号
- 在无样本验证下直接提高杠杆

### 11.1 第一版默认值

以下默认值用于启动 `Observe Mode`、`Shadow Mode` 和首轮参数研究。

**诚实声明：以下参数均为无理论推导的初始猜测值。** 它们不是从市场微观结构推导出的最优值，而是基于"保守、可积累样本、错误代价可控"原则选取的起点。每个参数的真实最优值只能通过 Observe Mode 和 Shadow Mode 的数据校准获得。

它们的目标不是最大化收益，而是：

- 先保证样本可积累
- 先保证错误代价可控
- 先保证影子模式能较快收敛

| 参数名 | 第一版默认值 | 用途 |
| --- | --- | --- |
| `liq_notional_threshold` | `2_000_000 USDT / 60s` | 过滤过小的强平噪音 |
| `liq_acceleration_ratio` | `1.0` | 当前窗口强平不大于前窗口时允许入场 |
| `deviation_threshold_bps` | `100` | 要求价格至少低于 `VWAP_15M` 1.0% |
| `entry_offset_bps` | `20` | 在触发价下方再挂 0.20% 的被动买单 |
| `order_ttl_sec` | `20` | 超过 20 秒未成交则撤单 |
| `hard_stop_bps` | `60` | 入场后再下跌 0.60% 触发价格硬止损 |
| `time_stop_sec` | `180` | 成交后 180 秒不修复则退出 |
| `stop_slippage_buffer_bps` | `30` | 对主动止损额外预留 0.30% 滑点预算 |

### 11.2 选择理由

`liq_acceleration_ratio = 1.0`

- 该参数控制衰减层的判定：当前窗口强平金额 > 前窗口 × 该倍率时阻断入场
- `1.0` 是最严格的设定：只要当前窗口不小于前窗口就阻断
- 首版选择最严格值，目的是在数据不足时宁可错过也不在加速强平中入场
- 后续可根据 Observe Mode 数据放宽到 `1.2` 或 `1.5`（允许轻微加速）

`liq_notional_threshold = 2_000_000 USDT / 60s`

- Binance 全市场强平流按 `1000ms` 推送快照，而且对同一 symbol 在 1000ms 内只保留最新一笔强平，因此该数据天然会低估真实连环清算强度
- 对 BTCUSDT 来说，若阈值设得过高，早期样本会过少；若设得过低，噪音事件会过多
- `2M / 60s` 是偏保守但仍能积累样本的起点

`deviation_threshold_bps = 100`

- 低于 `50-70bps` 时，很多事件更像普通波动而不是清算后的真空
- 高于 `150bps` 时，样本数量容易显著下降
- `100bps` 是首轮研究较稳的折中值

`entry_offset_bps = 20`

- 由于强平流本身是 `1000ms` 级别快照，直接按触发价挂单过于乐观
- 东京 VPS 部署后延迟降至 ~50ms，但仍保留 `20bps` 偏移作为安全缓冲
- 低延迟使得”检测到信号后再下单”成为可能，挂单精度显著提升
- 该值后续应结合影子模式的”错过率/回撤率”一起调

`order_ttl_sec = 20`

- 若 20 秒内都没有触发成交，说明这次下杀并未继续扩展到你的挂单位
- 将挂单长期留在盘口，容易把一次事件单变成脱离语境的普通抄底单

`hard_stop_bps = 60`

- 该值必须与 `entry_offset_bps` 分开理解：前者控制成交后的风险，后者控制挂单位置
- `60bps` 对 BTC 永续的短时修复模型来说足够紧，不至于让单次错误迅速放大

`time_stop_sec = 180`

- 若是清算后的修复性回抽，往往应在数十秒到数分钟内发生
- `180s` 能兼顾“给反弹一点时间”和“避免长期拿错单”

`stop_slippage_buffer_bps = 30`

- 价格硬止损只是触发阈值，不是最终成交价
- 在极端行情里，主动退出的真实滑点可能显著大于平时
- 东京 VPS 部署后，止损滑点预期从 ~0.3% 降至 ~0.05%，但首版仍保留 `30bps` 作为保守预算
- 后续可根据实际延迟分布（P50, P95, P99）收紧该值

## 12. 已冻结决策

当前版本冻结以下决策，后续编码按此执行：

### 12.1 历史强平数据路线

不依赖交易所提供“全市场历史强平数据”作为研究前提。

执行决策：

- 从 `Observe Mode` 第一天开始自建实时强平数据仓库
- 所有后续研究、回放、影子模式，默认基于自采集数据
- 若未来能拿到稳定可靠的外部历史强平数据，只作为补充，不改动首版架构前提

### 12.2 MVP 基准价

MVP 固定使用 `VWAP_15M` 作为基准价。

理由：

- 比 `EMA_15M` 更接近真实成交成本中心
- 计算复杂度仍低，适合首版实现
- 更符合”价格偏离近期真实成交成本”的策略解释

已知缺陷：

- VWAP 在强平过程中会被大量低价成交拉低 — 即基准价本身正在被检测事件污染
- 这意味着偏离率会被低估（实际偏离比计算值更大）
- 首版接受该缺陷，因为低估偏离率等于提高入场门槛，属于保守方向的误差
- 后续可考虑”冻结 VWAP”方案：在强平信号触发时锁定当时的 VWAP，不再随强平期间的成交更新

MVP 的 `VWAP_15M` 定义冻结如下：

- 使用最近 `15` 根 `1m` K 线计算 `Rolling VWAP`
- 首版不使用逐笔 `aggTrade` 在内存中连续累加 VWAP
- 系统重启后，可通过 K 线历史与实时 K 线流快速恢复基准价计算

### 12.3 MVP 止盈方式

MVP 固定采用 `单目标止盈`。

默认目标：

- 回归 `VWAP_15M` 或其附近预设回补区间

说明：

- 分批止盈保留到后续增强版本
- 首版优先保证可验证、可审计、可解释

### 12.4 实盘止损执行方式

MVP 允许止损使用主动退出。

执行原则：

- 入场必须保持 `Post-Only LIMIT`
- 止损时以风险控制优先，不强制要求继续做 Maker
- 若触发时间止损或价格硬止损，可使用更积极的退出方式完成平仓

理由：

- 止损阶段的首要目标是控制尾部风险，而不是优化手续费
- 若将止损也限制为被动挂单，可能在单边崩盘中扩大损失

### 12.5 止损滑点预算

MVP 必须将止损滑点视为真实成本，而不是异常值。

执行原则：

- 价格硬止损只定义“应退出”的逻辑阈值，不代表最终成交价
- 影子模式、测试网审计和实盘复盘都必须单独记录止损滑点
- 参数研究必须保留保守的止损滑点预算

## 13. 竞争环境分析

### 13.1 竞争事实

清算后的流动性真空并非只有本系统在参与。以下参与者也在争夺同一修复机会：

- **做市商**：延迟 < 1ms，在清算冲击发生的同时就能调整报价和挂单
- **专业套利机器人**：co-located 服务器，延迟 < 10ms，可在第一波冲击中抢先成交
- **其他清算猎手**：使用相同或类似的 `forceOrder` 数据源，策略同质化风险高

### 13.2 竞争对本策略的影响

- 参与者越多，回弹中可提取的利润越薄
- 快速参与者会优先消耗浅层流动性真空，留给慢速参与者的只有更深层的机会
- 如果同类策略普及，回弹速度可能趋势性加快，挤压慢速挂单的成交窗口

### 13.3 应对原则

本策略不试图在速度上取胜，而是接受以下定位：

- 只参与深层冲击（`entry_offset_bps` 确保挂单位远离快速参与者的争夺区）
- 只参与大规模事件（`liq_notional_threshold` 过滤小事件，大事件中真空更深、持续更久）
- 接受低成交率换取更优的入场位置

### 13.4 Observe Mode 额外观测维度

为评估竞争压力，Observe Mode 阶段应额外记录：

- 清算冲击后的回弹启动延迟（从最低点到首次回弹 > 10bps 的时间）
- 回弹深度（回弹幅度占冲击幅度的比例）
- 上述两个指标的时间趋势（是否在持续恶化）

若观测到回弹速度和深度在数周内持续显著下降，应视为竞争挤压的信号，需重新评估策略可行性。

## 14. 盈利模型量化

### 14.1 单笔期望值公式

```
E[PnL] = P(win) × AvgWin - P(loss) × AvgLoss - TotalFees
```

其中：

- `P(win)` = 成交后价格回归止盈目标的概率
- `P(loss)` = 成交后触发止损的概率
- `AvgWin` ≈ `deviation_threshold_bps` 回归幅度（~100bps）
- `AvgLoss` ≈ `hard_stop_bps` + `stop_slippage_buffer_bps`（60 + 30 = 90bps）
- `TotalFees` = Maker 入场 + 退出手续费

### 14.2 基于首版参数的盈亏平衡估算

假设条件（基于第一版默认值）：

| 项目 | 值 |
|------|-----|
| 止盈幅度 | ~100bps |
| 止损幅度（含滑点） | ~90bps |
| Maker 入场费 | 0.02% |
| Maker 止盈退出费 | 0.02% |
| Taker 止损退出费 | 0.04% |
| 盈亏比 | ~1.1 : 1 |

盈亏平衡胜率推导：

```
P(win) × 100 - (1 - P(win)) × 90 = P(win) × 4 + (1 - P(win)) × 6
```

解得：**盈亏平衡胜率 ≈ 51.6%**

### 14.3 VPS 部署对盈利模型的影响

东京 VPS 部署（延迟 ~50ms）对盈利模型产生以下量化影响：

| 指标 | 本地部署（300ms） | VPS 部署（50ms） | 影响 |
|------|-------------------|-------------------|------|
| 止损滑点预期 | ~0.30% | ~0.05% | 单笔最大亏损降低 |
| 可监控交易对 | 3-5 | 15-20 | 交易频率提升 ~4x |
| 多级建仓 | 不可行 | 可行（2-3 级） | 平均入场成本降低 |
| 盈亏平衡胜率 | ~51.6% | ~48%（滑点降低） | 门槛降低 |
| Orderbook 实时分析 | 不可行 | 可行 | 信号质量提升 |

VPS 成本约 10-30 USD/月，相对于策略可行性的提升，是最高性价比的投入。

### 14.4 对参数研究的指导意义

- 当前盈亏比偏低（~1.1:1），策略对信号质量高度敏感
- 提高盈亏比的方向：增大 `entry_offset_bps`（更深的入场位 → 更大的回归空间）或收紧 `hard_stop_bps`
- 但更深的入场位会降低成交率，更紧的止损会降低胜率 → 存在优化的上界
- VPS 部署后，多级建仓可降低平均入场成本，间接提高盈亏比
- Shadow Mode 的核心任务：验证不同参数组合下的实际胜率是否能超过盈亏平衡线

### 14.5 小账户手续费影响量化

以 1000 USD 账户、5% 保证金、3x 杠杆为例：

| 项目 | 值 |
|------|-----|
| 单笔名义价值 | 150 USD |
| Maker 入场费 | 0.03 USD |
| Maker 止盈费 | 0.03 USD |
| Taker 止损费 | 0.06 USD |
| 单笔来回（止盈路径） | ~0.06 USD |
| 单笔来回（止损路径） | ~0.09 USD |
| 手续费占止盈利润比 | ~4% |
| 手续费占止损亏损比 | ~6.7% |

结论：手续费占比可控，不是致命因素，但在盈亏比进一步压缩时会成为拖累。
