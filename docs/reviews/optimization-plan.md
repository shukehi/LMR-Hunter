# 第一性原理优化计划

## 1. 文档目的

本文档定义当前代码审查结论之后的优化顺序、边界、交付物与验收标准。

本文档回答的是：

- 先修什么，后修什么
- 哪些问题会直接污染策略有效性判断
- 在什么条件满足之前，不应继续调参数、不应推进 Shadow Mode、不应接入实盘

本文件是阶段性审查结论，不替代 [策略文档](../strategy.md)、[术语表](../glossary.md)、[架构文档](../architecture.md)、[风控文档](../risk-controls.md)、[开发计划](../development-plan.md) 和 [项目状态](../status.md) 这些长期规范文档。

## 2. 当前问题摘要

本次审查确认，项目文档层面的策略设计基本符合第一性原理，但当前代码实现仍未形成“可交易 edge 验证闭环”。

当前最关键的问题分为五类：

1. 样本定义错误
   - 当前把每一笔 BTC SELL 强平都直接当成一条研究样本
   - 一次连续踩踏会被重复计数，导致样本量虚高、统计显著性失真

2. 时间语义错误
   - 衰减判断使用本地当前时间而不是事件时间
   - 偏离率使用无时间戳的最新盘口中间价，无法保证与强平事件对时

3. 收益口径错误
   - 当前日报统计的是“后来是否涨回 VWAP”
   - 这不是“订单是否可成交、计入摩擦后是否赚钱”

4. 观测能力不足
   - 当前数据模型无法验证修复机制
   - 无法区分反弹来自 basis 收敛、做市商回补，还是随机波动

5. 数据完整性不足
   - 成交批量写入失败会直接丢样本
   - 样本损坏时系统仍会继续输出研究结论

## 3. 优化总目标

优化后的系统必须从“记录市场反弹”升级为“验证可交易错价”。

优化目标冻结如下：

- 样本单位必须是 `episode`，而不是单笔强平
- 所有关键特征必须按 `Exchange Time` 对齐
- 研究指标必须能回答“可否成交、成本后是否有正期望”
- 观测数据必须能支持策略失效归因
- 数据落库必须保证“失败可见、样本不静默丢失”

## 4. 优先级原则

本次优化遵循以下顺序：

1. 先修数据正确性，再谈统计结论
2. 先修事件定义，再谈阈值调优
3. 先修时间对齐，再谈信号质量
4. 先修收益口径，再谈策略 edge
5. 先补可解释观测，再谈扩展功能

冻结要求：

- 在阶段 3 完成前，不允许使用当前 `signals` 表结果做参数冻结
- 在阶段 5 完成前，不允许使用当前 `daily_report.py` 输出作为策略有效性证据
- 在阶段 6 完成前，不允许推进 Shadow Mode 参数扫描

### 阶段 1：修复数据完整性与失败可见性

目标：

- 消除成交样本静默丢失
- 让数据损坏成为显式风险事件，而不是后台噪声

要做的事：

- 重写 `flush_trades()` 批量写入流程
- 在写库成功前，不从内存队列永久移除样本
- 写入失败时将 batch 回队列，或写入隔离区文件
- 为批量写入增加重试次数、退避策略、失败计数
- 当连续失败超过阈值时，写入 `risk_events` 并将系统状态标记为 `DEGRADED` 或 `HALTED`
- 增加数据完整性计数器：
  - 入队笔数
  - 成功落库笔数
  - 重试笔数
  - 丢弃笔数

交付物：

- `writer.py` 容错改造
- 写入失败的隔离策略文档
- 数据完整性统计字段
- 对应单元测试

验收标准：

- 人为注入 SQLite 写入失败时，样本不会静默丢失
- 连续失败会产生明确告警与风控事件
- 恢复后数据计数可对齐
- `discarded_trades = 0` 才允许进入下一阶段

### 阶段 2：修复时间语义，所有特征以事件时间对齐

目标：

- 消除“把链路延迟误判为市场结构”的问题
- 让衰减层和偏离层具备可解释性

要做的事：

- 为盘口中间价缓存增加时间戳，至少保留：
  - `depth_event_ts`
  - `depth_recv_ts`
  - `mid_price`
- 将 `snapshot()` 改为显式接收锚点时间：
  - 推荐接口：`snapshot_at(event_ts_ms, recv_ts_ms=None)`
- 强平窗口聚合与衰减判断统一基于 `event_ts`
- 偏离率计算只允许使用满足 freshness 条件的盘口快照
- 若没有足够接近 `event_ts` 的盘口，则该次偏离率标记为不可用，不输出研究样本
- 为乱序、延迟、静默断流场景增加测试用例

交付物：

- `FeatureCalculator` 时间语义重构
- 盘口 freshness 规则
- 事件时间对齐测试

验收标准：

- 同一组强平事件在不同本地处理延迟下，输出的 `liq_accel_ratio` 一致
- 偏离率的输入价格能够追溯到与事件对齐的盘口快照
- 对时失败时系统拒绝生成研究样本，而不是继续输出错误特征

### 阶段 3：引入 episode 模型，修复研究样本定义

目标：

- 将研究单位从“单笔强平”改为“连续冲击 episode”
- 避免一次踩踏事件被重复计数

要做的事：

- 定义 `liquidation episode`：
  - 起点：窗口内强平规模首次超过噪声门槛
  - 持续：连续窗口仍存在同方向冲击
  - 终点：强平金额回落到终止阈值，或超过最大持续时长
- 新增 episode 聚合对象，至少包含：
  - `episode_id`
  - `start_event_ts`
  - `end_event_ts`
  - `liq_notional_total`
  - `liq_peak_window`
  - `liq_accel_ratio_peak`
  - `min_mid_price`
  - `pre_event_vwap`
  - `max_deviation_bps`
- 只有 episode 结束后，才生成一条标准研究样本
- `signals` 表要么升级为事件级样本表，要么新增 `liquidation_episodes` 表
- 原始强平事件与 episode 建立可追溯关联

交付物：

- `episode builder`
- 新的 episode 持久化结构
- 样本口径迁移说明

验收标准：

- 一次连续踩踏只生成一个 episode 级样本
- episode 可回放到原始强平事件集合
- 样本数量与人工标注的 episode 数量大体一致

### 阶段 4：补齐修复机制观测数据

目标：

- 让系统不仅能看到“反弹是否发生”，还能看到“为什么发生”
- 为失效检测提供因果线索

要做的事：

- 扩展持久化层，新增或补充以下观测字段：
  - 盘口深度摘要
  - 冲击时的买盘承接量
  - 永续 vs 指数/现货价差快照
  - 回弹启动延迟
  - 回弹深度
  - episode 期间最低点、最高回补点
  - 冲击后的盘口恢复速度
- 设计 episode outcome 表，记录：
  - `rebound_started_at`
  - `rebound_start_latency_ms`
  - `rebound_depth_bps`
  - `mae_bps`
  - `mfe_bps`
  - `basis_at_impact`
  - `basis_at_recovery`

交付物：

- schema 扩展
- 观测字段写入逻辑
- outcome 回填逻辑

验收标准：

- 每个有效 episode 都能生成 outcome 记录
- 至少能回答以下问题：
  - 反弹是否发生
  - 反弹延迟多久发生
  - 反弹深度多大
  - basis 是否显著收敛
  - 盘口是否出现承接恢复

### 阶段 5：重写研究报告，改为可交易口径

目标：

- 把“会不会回弹”改成“能否赚钱”
- 让参数建议基于净期望，而不是图形上的回归

要做的事：

- 停用当前“最高价是否回到 VWAP”的主结论口径
- 新研究口径至少包含：
  - 订单是否可能成交
  - 挂单生效前的延迟惩罚
  - Maker 入场假设
  - 止盈、价格止损、时间止损
  - 手续费与最小滑点预算
  - episode 级净 PnL 分布
- 报表输出改为以下核心指标：
  - fill rate
  - net win rate
  - avg win / avg loss
  - expectancy per trade
  - MAE / MFE 分布
  - 对参数敏感的失效区间
- 当前”回到 VWAP 的比例”只保留为辅助观测字段，不得作为参数建议主依据
- **按盘口分组统计**：不同交易时段（亚盘 / 欧盘 / 美盘）的流动性结构差异显著，
  混合统计会稀释甚至失真结论。分组规则从 `start_event_ts` 推算，无需改 schema：
  - 亚盘：00:00–07:00 UTC
  - 欧盘：07:00–13:30 UTC
  - 美盘：13:30–21:00 UTC
  - 夜盘：21:00–00:00 UTC（低流动性，可单独标注）
  - 参考：[市场时段文档](../market-sessions.md)

交付物：

- 新版 `daily_report.py` 或新的 `research_report.py`
- 净期望统计口径说明
- 参数建议模板
- 按盘口分组的分层统计报告

验收标准：

- 报告主结论基于 episode 级可交易模拟结果
- 所有参数建议都能追溯到净收益统计，而不是价格形态
- 报告能够显式区分”有反弹但不可交易”和”可交易且净盈利”
- 报告按亚 / 欧 / 美盘分组输出统计，不允许将三个时段混合作为唯一结论

### 阶段 6：建立进入 Shadow Mode 的前置门槛

目标：

- 在研究口径正确之前，不进入更高复杂度阶段

要做的事：

- 定义 Shadow Mode 前置门槛：
  - 数据完整性稳定
  - episode 样本定义稳定
  - 时间语义正确
  - 观测字段足以支持失效归因
  - 可交易研究报告已运行至少若干天
- 将这些门槛写入开发计划与风控文档

交付物：

- Shadow Mode 准入清单
- 更新后的阶段门禁文档

验收标准：

- 未满足门槛时，系统只能停留在 Observe Mode
- 门槛满足后，才允许实现虚拟挂单和净收益回放

## 6. 与当前发现的映射关系

| 审查问题 | 根因类别 | 对应阶段 |
| --- | --- | --- |
| 写库失败直接丢成交样本 | 数据完整性 | 阶段 1 |
| 每笔强平都被当成独立统计样本 | 样本定义 | 阶段 3 |
| 衰减判断使用本地当前时间 | 时间语义 | 阶段 2 |
| 偏离率基于未对时盘口 | 时间语义 | 阶段 2 |
| 回弹研究指标不是可交易收益指标 | 收益口径 | 阶段 5 |
| 数据模型缺少关键观测 | 观测能力 | 阶段 4 |

## 7. 推荐执行顺序

推荐按以下顺序推进：

1. 阶段 1：先止血，确保样本不丢
2. 阶段 2：修正时间语义，保证特征可信
3. 阶段 3：修正 episode 口径，保证统计不重复计数
4. 阶段 4：补齐观测字段，保证失效可归因
5. 阶段 5：重写研究报告，保证结论与净收益相关
6. 阶段 6：冻结准入门槛，再推进 Shadow Mode

禁止倒序推进：

- 不允许先做参数扫描再修时间语义
- 不允许先做 Shadow 撮合再修 episode 口径
- 不允许基于当前日报调整 `deviation_threshold_bps`

## 8. 里程碑定义

### 里程碑 M1：数据可信

满足条件：

- 成交样本无静默丢失
- 批量写入失败可重试、可告警、可审计

### 里程碑 M2：特征可信

满足条件：

- 衰减层与偏离层全部按事件时间对齐
- stale 盘口不会参与偏离率计算

### 里程碑 M3：样本可信

满足条件：

- 所有研究样本均为 episode 级
- 同一冲击不会被重复统计

### 里程碑 M4：结论可信

满足条件：

- 报告主指标基于可交易净收益
- 关键观测足以支持 edge 验证和失效归因

## 9. 优化完成后的预期状态

完成本计划后，系统应能回答以下第一性原理问题：

1. 是否真的出现了异常、非自愿抛售
2. 该抛售是否在事件结束后留下了可交易错价
3. 错价是否足够深，能覆盖手续费、滑点、延迟和止损成本
4. 修复是由哪些驱动力推动的
5. 当修复失效时，系统是否能自动识别并停止推进

只有在以上问题都能被数据回答后，项目才有资格从“市场观察器”进入“交易系统”阶段。
